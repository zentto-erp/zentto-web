## 1. Contexto — 12 pain points verificados

Inventario de dolor recogido del código actual del CRM (rama `developer` antes de este milestone). Cada punto cita el síntoma observado en `apps/crm` y en los SPs de `schema crm`.

1. **Listas con `<table>` nativa y paginación rota.** Las pantallas `LeadsListPage` y `CampaignsListPage` renderizan tablas HTML crudas, sin virtualización, con columnas fijas y paginación en servidor pero UX de scroll infinito a medias. Rompe la [regla crítica](../../.claude/projects/d--DatqBoxWorkspace-DatqBoxWeb/memory/feedback_no_html_tables.md) de usar `<ZenttoDataGrid>`.
2. **Lead y Deal mezclados en una misma tabla.** `crm.Lead` carga prospect frío + oportunidad con pipeline, lo que se refleja en una UI confusa donde el usuario no distingue fase de prospecting vs. fase de venta. Ver ADR-CRM-001 decisión 2.
3. **No existe Contact/Company como entidad.** No hay pantalla de contactos; los datos viven como strings `ContactName/Email/Phone` en Lead. Impide deduplicación y relaciones N:N.
4. **Pipeline Kanban con drag-and-drop frágil.** El componente hace mutaciones optimistas sin rollback; si el SP falla, la tarjeta queda en la columna equivocada y solo se descubre al refrescar.
5. **Filtros con dropdowns dispersos.** Cada lista tiene sus filtros en una toolbar distinta, sin persistencia, sin combinaciones guardadas. No existe concepto "vista guardada" (ya abierto en CRM-108 #382).
6. **Detail pages abren en ruta nueva.** Cada click en Lead navega a `/crm/leads/:id`, forzando un viaje ida-y-vuelta. Competencia (HubSpot, Linear, Attio) resuelve con drawer lateral.
7. **Activity timeline inconsistente.** `Activity`, `CallLog` y `LeadHistory` viven en 3 timelines distintos, no se pueden filtrar cruzados ni mostrar en una sola vista cronológica.
8. **Sin command palette ni atajos de teclado.** El operador debe ir al menú, al botón, al submenú. No hay `Cmd+K` ni shortcuts para "nuevo lead", "buscar empresa", "marcar won".
9. **Densidad no configurable.** La grilla es siempre la misma altura. Operadores power-user pierden pantalla; usuarios ocasionales no entienden el layout compacto.
10. **Responsive roto bajo 1280px.** Columnas se cortan, kanban colapsa, drawer de filtros se superpone al contenido.
11. **Empty states sin acción.** "No hay leads" no ofrece botón directo para crear; forza ir a otro lado del menú.
12. **SQL Server desnivelado.** Solo 5 de 14 tablas CRM existen en `sqlweb-mssql`. Clientes en SQL Server no pueden usar el módulo a paridad con PG. Ver ADR-CRM-001 consecuencia #3.

Estos 12 puntos se mapean 1:1 a los 14 issues del milestone CRM Redesign · Fase 1 (ver sección 7).

## 2. Referencias analizadas

### Productos líderes

- **HubSpot Smart CRM** — modelo Contact / Company / Deal con relaciones N:N, timeline unificado, properties extensibles, vistas guardadas como tabs superiores, drawer derecho con tabs internos.
- **Linear** — command palette `Cmd+K` omnipresente, atajos `C` crear, `G then I` ir a inbox, filtros combinables con operadores, densidad controlable (cozy/compact/comfortable).
- **Pipedrive** — Deal-first, pipeline visual, velocity analytics, coaching tips. Flujo de conversión Lead → Deal explícito.
- **Attio** — RecordTable con scroll horizontal virtualizado, columnas redimensionables/reordenables inline, chips tipo-aware (email, phone, url, currency), drawer derecho con expansión a pantalla completa.

### Design systems

- **Shopify Polaris IndexTable** — patrón canonical de lista de recursos con bulk actions, resource name, loading state, empty state.
- **Atlassian Design System** — Drawer lateral con stack de niveles, Board (kanban) con columnas fijas y virtualización por columna.
- **GitHub Primer CommandBar** — patrón de command palette con secciones, acciones, navegación, búsqueda fuzzy.
- **Shadcn cmdk** — open source, accesible, base para nuestra paleta.
- **IBM Carbon** — tokens de spacing (4/8/16/24/32), jerarquía tipográfica, motion.
- **Base Web (Uber)** — Table con scroll virtualizado estable y columnas sticky.

## 3. Principios de diseño (6)

1. **Data-first.** La pantalla principal es la lista. Herramientas (filtros, búsqueda, acciones) giran alrededor de los datos. Todo chrome (headers, sidebars) es secundario.
2. **Keyboard-first.** Todo se opera con teclado. `Cmd+K` paleta, flechas navegan, `Enter` abre drawer, `E` edita, `N` nuevo, `/` busca, `F` filtra. Mouse es equivalente, no superior.
3. **Densidad escalable.** 3 modos: compact (32px row), cozy (40px, default), comfortable (52px). Usuario elige, preferencia persiste por usuario en `user.Settings.crm.density`.
4. **Drawer, no navegar.** Click en fila abre `RightDetailDrawer`. Expandible a pantalla completa si se quiere más contexto. Doble click o `Shift+Enter` navega como hoy (fallback).
5. **Filtros y vistas guardadas son ciudadanía primera clase.** Tabs superiores son vistas (`All contacts`, `My contacts`, `Hot leads`, ...). Tab `+` crea vista nueva. Filtros combinables, persistidos (CRM-108 #382).
6. **Empty states accionables.** Cada pantalla vacía tiene un CTA primario, un CTA secundario y un link a docs. "No hay contactos" → `[Crear contacto]` `[Importar CSV]` `Cómo funciona →`.

│ nav    ├─────────────────────────────────────────────────────────┤

### 4.2 Seis componentes nuevos (scope shared-ui)

1. **`<ZenttoRecordTable>`** — successor de `<ZenttoDataGrid>` orientado a records tipo-aware. Features:
   - Virtualización filas y columnas (react-virtual).
   - Columnas redimensionables, reordenables, sticky.
   - Chips tipo-aware (`email`, `phone`, `currency`, `date`, `status`, `user`, `relation`).
   - Bulk select con `Shift+Click`, `Cmd+Click`, `Cmd+A`.
   - Inline edit (doble click celda) opcional por columna.
   - Densidad configurable.
   - Export a CSV/XLSX vía `@zentto/shared-export`.
   - Accesibilidad: `aria-rowindex`, `aria-colindex`, navegación con flechas.

2. **`<CommandPalette>`** — `Cmd+K` global. Secciones:
   - **Navegación** (ir a Contactos, Deals, Pipeline, ...).
   - **Acciones** (Crear contacto, Crear deal, Nueva llamada, Marcar won, ...).
   - **Búsqueda** (fuzzy sobre contacts/companies/deals vía endpoint `/v1/crm/search`).
   - **Vistas recientes**.
   - Shortcut badges a la derecha. Footer con `↑↓` navegar, `Enter` ejecutar, `Esc` cerrar.
   - Base: `cmdk` + tokens Zentto.

3. **`<RightDetailDrawer>`** — panel lateral 40% del ancho (resizable, min 480px, max 960px). Stack de niveles: abrir un relacionado (ej. Company desde Contact) empuja el anterior. Breadcrumb interno. Botón `⛶` expande a fullscreen. Tabs internos: Overview / Actividad / Notas / Archivos / Relaciones.

4. **`<QuickCreate>`** — modal pequeño (420px) con formulario mínimo: Contact (nombre + email + phone), Company (nombre + website), Deal (nombre + contact + value + pipeline). Se abre desde CommandPalette o con `N`. Crea y vuelve a la lista con el nuevo record seleccionado.

5. **`<ActivityTimeline>`** (unificado) — consolida `crm.Activity` + `crm.CallLog` + `crm.LeadHistory` + notas + emails en UN solo timeline con filtros por tipo. Icono por tipo, agrupado por día, click expande detalle inline.

6. **`<PipelineKanban>` v2** — sobre `crm.Deal` (ya no Lead). Columnas = stages. Virtualización por columna. Drag con `react-dnd`, con optimistic UI + rollback si SP falla. Filtros por owner/valor/fecha. Totales por columna. Atajos `1..9` para mover selected card a stage N.

### 4.4 Atajos de teclado

| Shortcut | Acción |
|----------|--------|
| `Cmd+K` | Command palette |
| `N` | Nuevo (contextual: contact si estoy en contacts, deal en deals) |
| `/` | Enfocar búsqueda |
| `F` | Abrir panel de filtros |
| `E` | Edit record seleccionado |
| `Enter` | Abrir drawer |
| `Shift+Enter` | Navegar a detalle fullscreen |
| `Esc` | Cerrar drawer/modal/paleta |
| `↑/↓` | Navegar filas |
| `←/→` | Navegar columnas (cuando hay inline edit) |
| `Cmd+A` | Seleccionar todos (visible) |
| `1..9` | Mover tarjeta Kanban a stage N |

### 4.5 Responsive

- **≥ 1280px**: layout completo con side nav + tabla + drawer side-by-side.
- **768–1279px**: side nav colapsa a iconos, drawer se vuelve overlay.
- **< 768px**: navegación con drawer inferior tipo mobile, tabla pasa a list cards, Kanban pasa a lista por stage.

Breakpoint primario del diseño: **desktop (1440px)**. Ver pregunta abierta en §8.

### 4.6 Accesibilidad

- Todo interactivo alcanzable por teclado.
- `aria-label` explícito en iconos, `aria-describedby` en campos con error.
- Contraste mínimo WCAG AA.
- Focus ring visible con token `--color-focus-ring`.
- Soporte `prefers-reduced-motion` en transiciones del drawer y kanban.
- Screen reader: `aria-live="polite"` en notificaciones, `role="grid"` en RecordTable.

## 5. Criterios de aceptación (≥ 20)

- [ ] Listas CRM no usan `<table>` nativa; todas usan `<ZenttoRecordTable>` o `<ZenttoDataGrid>` de shared-ui.
- [ ] `<ZenttoRecordTable>` publicado en `@zentto/shared-ui` ≥ v1.3.0.
- [ ] Virtualización soporta 10k filas sin jank (> 50fps en scroll).
- [ ] Columnas redimensionables, reordenables, sticky primera columna.
- [ ] Chips tipo-aware para email/phone/currency/date/status/user/relation.
- [ ] CommandPalette disponible con `Cmd+K` en todas las rutas `/crm/*`.
- [ ] CommandPalette incluye ≥ 20 acciones core + search + recientes.
- [ ] RightDetailDrawer abre con `Enter` o click en fila.
- [ ] RightDetailDrawer soporta stacking de ≥ 3 niveles con breadcrumb.
- [ ] QuickCreate accesible desde paleta y con `N` por ruta contextual.
- [ ] ActivityTimeline unifica Activity + CallLog + LeadHistory + Notas.
- [ ] PipelineKanban sobre `crm.Deal`, no sobre Lead.
- [ ] Kanban con optimistic UI + rollback en fallo de SP.
- [ ] Densidad compacta / cozy / comfortable, persistida por usuario.
- [ ] Vistas guardadas con tabs superiores (consume CRM-108 SavedView API).
- [ ] Empty states con CTA primario + secundario + link docs.
- [ ] Shortcuts documentados en `docs/wiki/DESIGN-SYSTEM.md` sección "CRM".
- [ ] Responsive probado a 1440 / 1280 / 1024 / 768 / 390 px.
- [ ] Accesibilidad: auditoría axe sin críticos.
- [ ] Observabilidad: eventos `obs.event('crm.table.load', ...)`, `crm.drawer.open`, `crm.palette.action`.
- [ ] Docs en `docs/wiki/16-crm-entities-model.md` + `apps/crm/DESIGN.md` cerrados.
- [ ] SQL Server nivelado (Contact / Company / Deal existentes en ambos motores).

## 6. Riesgos e integración

### Riesgos de diseño
- Introducir `<ZenttoRecordTable>` nuevo sin deprecar `<ZenttoDataGrid>` crea deuda. Decisión: coexisten; RecordTable es superset, DataGrid sigue para ERP clásico.
- Drawer stacking > 3 niveles confunde; se limita a 3 hard.
- Atajos de teclado colisionan con navegador. Se filtra `Cmd+K` dentro del shell, no se intercepta si foco está en campo.

### Integration flags
- Cambios a shared-ui ⇒ impactan TODAS las apps consumers. **Exige integration review** → ver documento hermano `2026-04-19-crm-integration.md`.
- Cambios a `crm.Deal` ⇒ romper PipelineKanban hasta backfill. Mitigar con `vars.CRM_NEW_UI` opt-in.
- API nueva Contact/Company/Deal ⇒ observabilidad, audit log, multi-tenant, RBAC.

## 8. Pregunta abierta

**Breakpoint primario**: ¿diseñamos pensando en desktop (1440px) con mobile como degradación, o tablet (1024px) como ancla? La decisión afecta RecordTable (columnas visibles por defecto), Drawer (overlay vs side-by-side), Kanban (3 cols vs 5 cols).

**Recomendación del designer**: **desktop 1440px primary**, tablet y mobile son degradación progresiva. Los usuarios del CRM son power-users, no consumers en movimiento.
